# G2 Design Principles

## Package-driven development

Every implementation change belongs to exactly one ordered package:

```text
P0001, P0002, P0003, ...
```

A package is the atomic solution-version unit. It may update memory, requirements, source, deploy artifacts, tests and diagnostics.

Rollback is also a new forward-moving package.

## Requirements do not mirror code structure

Requirements are organized by intent and behavior:

```text
Epic -> Feature -> User Story -> Package
```

Code is organized by runtime/environment:

```text
src/mac
src/ha
src/shelly
src/shared
```

A feature or package may span multiple scripts, devices and environments.

Example:

```text
A package can update a Mac spot-price forecast service and a Shelly client that consumes that forecast.
```

## Source and deploy are different layers

Development source should be optimized for comprehension, tests and Codex work.

Deploy artifacts should be optimized for their target runtime constraints.

For Shelly this means:

```text
source files      logical responsibility
built script      one complete script per runtime role
deploy chunks     generated numeric byte-size chunks
```

Do not let Shelly chunk structure define source architecture.

## Generated deploy artifacts

Deploy artifacts that can be generated deterministically should be generated by tools.

Codex should not manually improvise numeric deploy chunks.

For Shelly code packages, Codex should modify `src/`, run tests, run build tooling, validate `dep/s/`, and then commit source plus generated deploy artifacts when the package allows.

## Determinism and observability

Prefer deterministic logic, explicit state and visible failures.

Avoid broad defensive normalizers that hide bugs. Use explicit fallback behavior for missing telemetry, unavailable services or stale external inputs.

## Local hardware safety

Devices that directly control hardware should have conservative local fallback/default behavior.

Remote planners and optimizers may improve comfort/cost, but local hardware controllers must remain safe if external control disappears.

## Separation of planning and actuation

Planning can happen on Mac or Home Assistant.

Slow equipment should not be constantly replanned by fast loops.

Shelly edge code should stay small, deterministic and focused on local control/application.

## Device identity and replacement

Stable G2 identity is the logical device name.

Physical Shelly id, IP and hardware instance are replaceable.

Device replacement should normally require changing only the physical-to-logical registry mapping.

## Installer truth

GitHub deploy artifacts define desired state.

Local Shelly state and readback define actual state.

A single persistent text component may store the last verified device target version.

KVS is runtime/cache/bootstrap state and must not be the only installer truth.

## Testability

Every package must define test cases and verification commands.

Codex must run or explicitly report why it could not run them.
